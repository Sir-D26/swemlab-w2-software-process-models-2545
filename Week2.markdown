Step 1: Selected Software Process Model
The most suitable software process model for developing the Campus Event Finder mobile application is the Agile Model.
Justification
The Agile model is appropriate because it supports iterative development, flexibility, and continuous user feedback. The project description indicates that while the core features are known, the organization expects additional features in the future based on student feedback. Agile allows developers to release a working version of the application early and improve it through multiple development cycles.
The project also has a small budget and a 6-month timeline, which makes Agile advantageous because it prioritizes delivering working software quickly and adapting to changes efficiently.
Therefore, Agile is the best fit since it enables the team to build the core application first and gradually add improvements such as event registration, map integration, or social sharing based on user needs.
________________________________________
Step 2: Development Using the Generic Process Framework (Agile)
In Agile development, the Generic Process Framework activities occur repeatedly in short cycles called sprints. Each sprint delivers a small, functional part of the system.
________________________________________
1. Communication
Communication occurs continuously throughout the project between the development team and stakeholders.
Activities
•	Meetings with the student organization and university representatives.
•	Gathering requirements for the core application features.
•	Creating user stories describing what students want the app to do.
•	Collecting feedback from students after early versions are released.
Deliverables
•	User stories
•	Product backlog (list of desired features)
Example user stories:
•	As a student, I want to view upcoming campus events so I can know what activities are happening.
•	As a student, I want to filter events by category so I can find events that interest me.
•	As a student, I want to mark events as interested so I can receive reminders.
________________________________________
2. Planning
Planning occurs at the beginning of the project and at the start of each sprint.
Activities
•	Define the 6-month release goal.
•	Break the project into short sprints (2–3 weeks).
•	Prioritize the most important features first.
Example sprint plan:
Sprint 1
•	Connect to university calendar API
•	Display event list
Sprint 2
•	Implement category filtering
Sprint 3
•	Add “Interested” feature
Sprint 4
•	Implement reminder notifications
Sprint 5
•	Improve user interface
Sprint 6
•	Testing and bug fixes
Deliverables
•	Sprint backlog
•	Project timeline
•	Task assignments
________________________________________
3. Modeling
Modeling focuses on lightweight system design rather than extensive documentation.
Activities
•	Design the user interface layout.
•	Create wireframes for main screens.
•	Define system architecture for the mobile application.
Deliverables
•	UI wireframes and mockups
•	Basic system architecture diagrams
Example system structure:
•	Mobile application interface
•	University calendar API
•	Local database for storing “Interested” events
________________________________________
4. Construction
Construction involves coding and testing the application.
Activities
•	Develop the mobile app features.
•	Integrate the university calendar API.
•	Implement filtering and reminder functionality.
•	Perform testing such as:
o	Unit testing
o	Integration testing
o	User interface testing
Deliverables
•	Working software increments
•	Tested application modules
•	Updated source code repository
________________________________________
5. Deployment
Deployment involves releasing the application to users and collecting feedback.
Activities
•	Release beta versions to students.
•	Monitor bugs and performance issues.
•	Gather feedback for improvements.
•	Update the backlog with new feature requests.
Possible new feature requests:
•	Event registration
•	Map integration
•	Social media sharing
Deliverables
•	Beta releases
•	User feedback reports
•	Updated product backlog

Step 3: Alternative Model (Less Suitable)
An alternative software process model that would be less suitable for this project is the Waterfall Model.
Explanation
The Waterfall model follows a strict sequential development process, where each phase must be completed before the next one begins. This model requires all system requirements to be clearly defined at the start of the project.
However, the Campus Event Finder application expects future changes and feature requests from students, such as event registration and map integration. Since Waterfall does not easily accommodate changes once development has started, modifying the system later could be time-consuming and costly.
Additionally, in the Waterfall model, users typically see the working system only after development is complete. This limits opportunities for early feedback, which is important for improving the usability of a student-focused mobile application.
Therefore, the Waterfall model would be a poor fit because it lacks the flexibility and iterative development needed for this project.
________________________________________
Conclusion
The Agile software process model is the most appropriate approach for developing the Campus Event Finder application because it allows iterative development, frequent user feedback, and adaptability to changing requirements. By progressing through the Generic Process Framework activities repeatedly in short sprints, the development team can deliver a functional product within the 6-month timeline while continuously improving the application based on student feedback.
